# 運賃マスタの作成

運賃マスタとなると、さすがに業務的なロジックに依存してくるので、汎用的なフォームでの入力が難しくなる。  
まずはスキーマを定義してから、prompt でAIに指示しながら作成する。  

## prompt

とりあえずスキーマを定義したので、AI に説明しはじめる。

```markdown
packages/shared/src/types/tariffs.ts を見てくれ。  
今までのマスタ系コンテナと少々構造が違う事がわかるだろ。
---
今までと違って業務ロジックに依存するので、汎用的な解決は出来ない。
まずは `/masters/tariffs` というURLルーティングと、専用ページ(工事中)を作って、サイドメニューに登録してみてくれ。  
運賃っぽいアイコンでたのむ。
```

まずはベースが出来た。

### Stepper の導入

UIはウィザード形式にするので、QStepper による画面遷移を行う。  

```markdown
続いて作成した TariffsPage.vue に QStepper を導入する。
- STEP1: 運賃表のバージョンと距離範囲を設定する。
    - TariffRange のレコード数とバージョン(TariffsItem の id になる)を決めるステップ
- STEP2: 作成したレンジの各数値を入力する。
- STEP3: 確認画面(保存するか入力に戻るか)
こんなイメージで画面遷移だけ準備してみてくれ。中身は「STEP1」とか「STEP2」とかでOK。
```

それっぽいのが出来たけど、仰々しい実装もあるので修正させる。

```markdown
各ステップの icon だけど、直接指定できるので…
route, edit-note, task-alt という単純なキー指定にしてくれ。
---
ところでSTEP1だけ edit アイコンになってるのは、何故かな？
であれば、STEP1はアイコンなしでOKす。表示されることがない。
---
i18n だけど、あそこまで深い構造にしなくてOK。
tariffs.step1.title くらいの構造かな。なにせ3ステップだから。
```

これで画面遷移のベースまで作成できた。

### Store の作成

他のマスタストアを基本的に同じ構造だが、いくつかメソッドを追加することになるだろう。  

```markdown
続いて Tariffs 専用の store を用意してほしい。  
基本的な構造は、他のマスタStoreと同じであると考えてOKだ。
```

Functions 側をデプロイしないと、SPA側が死ぬという罠があったけど、とりあえずこちらもベースは出来た。

### STEP1のUI

STEP1は運賃表のバージョンと、レンジ(レコード数)を決める。  
1 ~ 100kmまでなら〇〇円ですみたいな範囲を先に決めることで、運賃設定の設計を理解しやすく？みたいなイメージだが。  
バージョンは運賃が時々変更されるわけで、過去の運賃表も残しておく必要があるって話である。  
期間限定運賃もあるやろう。

```markdown
もういちど `packages/shared/src/types/tariffs.ts` を確認してほしい。  
いくつかフィールドを増やしてある。
---
STEP1では以下の入力を想定している。
- Version 自動採番(レコードがなければ1で、あればレコード数+1)
- 運賃表の名前(必須)。ラベルみたいなもの。
- 運賃表の備考(必須ではない)。textarea 
- Range の設定
    - minKm と maxKm を確定する。
    - 1km ～ 20km、21km ～ 40km、... といったイメージでレコードの件数を確定するのが目的である。
    - バリデータは少々面倒かもしれないので、あとで考える。
    - 当然だけど、+ ボタンとかで最大100くらいまで増やせる。(実際には10レコードくらいやろう)
- components/masters/TariffsStep1.vue みたいなコンポーネントとして切り出しておく。
    - 単純に小さくして見やすくするのが目的であり、再利用性はないはず。
こんなイメージで入力フォームのモックを作れそうか？
```

勝手に動きまでついてる。
大したものだが、あってるのかは謎。  
まずはデザインの調整から入る。

```markdown
まずは、備考欄を距離レンジの下に配置してみてくれ。主役は距離レンジなんでね。
---
続いてフォーム全体を QCard による構成に変更してみてくれ。横幅に余裕がありすぎるので全体的中央寄せにしようかと思ってる。
---
「次へ」ボタンでバリデートもする予定なので、QFormも追加しておいてくれ。
---
QForm は import しなくても使えるやろ？
```

悪くないけど、画面的に余裕があるのでダイアログフォームに変更しよう。

```markdown
この QStepper で組んだ、STEP1 ～ STEP3 を QDialog を使った、ポップアップフォームにしたいけど、出来そうか？
---
STEP1 から STEP2 へデータを引き継ぐ必要があるけど、この機能は再利用せいもないので、わざわざ Store を用意する必要ない。
そういった観点から、TariffsDialog.vue でリアクティブな変数を初期化して、provide/inject する方がスマートなのではないかね？
```

デザインは微妙だけど、設計は理想に近づいてきた。  


### STEP2のUI

STEP1で作成した ranges に対して、金額を入れるのがSTEP2である。

```markdown
STEP2 は STEP1 で設定した ranges に対して、金額などを入力するフォームになる。
- 基本料金 baseFare は、そのレンジの最低料金となる。
- 加算単位 unitKm は、〇kmごとに加算しますという単位であって、デフォルト 1 である。たぶん、実質1である。
- 加算料金 unitFare は、unitKmごとに加算する料金である。
「○○km ～ ○○km 基本料金 [○○]円、[〇]kmごとに[○○]円加算」
```

### ロジックの整理

細かく指定せずに、とりあえず動くようにという指示ですすめると、いい加減で汚いコードになるようである。  
それでも動くようにしてるだけ大したものではあるが、1つずつ修正させていくとしよう。

```markdown
- components/masters/TariffsDialog.vue
- components/masters/TariffsForm.vue
- composables/tariff-draft.ts
QStepper を利用するために、step が文字列であるものの、片や数値で取り扱っている。  
公式サンプルにもあるように、この機能を使うには数値で管理する方が効率が良いのではないか？
```

続いて、minKm を自動算出するロジックの修正。

```markdown
minKm は基本的に自動算出できるはずである。
現状は watch(draft.value.ranges,xxx) で実装している。
これを model-value + computed + @update に変更できる気がする。
- computed を使って、minKm を自動算出する。
    - ranges.length === 0 であれば、1固定
    - 一つ前 ranges[i - 1] の maxKm + 1 を返せばOKなはず。
- したがって、readOnly でも問題ないのではないか？
実装可能か確認してみてくれ。
---
model-value には computed を使うべきだと言っているのだが？
```

続いて、baseFare も同様のロジックに変更させる。

```markdown
baseFare も同じ理屈で model-value + computed + @update に出来るのでは？
updateMaxKm って必要か？
---
baseFares が利用されるのは STEP2 と STEP3 だろ。
minKm が更新されるのは STEP1 だけだろ。
prev.minKm を computed 見たところで、結果は同じではないか？
---
updateMaxKm って、minKms computed と処理が被ってるやろ？
---
そもそも、minKm で @update を使って更新できるのでは？
となると、updateMaxKm で行うべきは、minKms computed を使って更新するべきなのでは？
```

これはあっさりできた。2回目だからだな。
